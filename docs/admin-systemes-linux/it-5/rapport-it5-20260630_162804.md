## Causes et risques traites

| Cause ou risque initial | Effet possible | Reponse appliquee |
| --- | --- | --- |
| Sauvegarde non testee | Impossible de prouver une reprise apres incident | Sauvegarde rsync, archive tar, restauration dans /tmp et checksum |
| Integrite non controlee | Archive modifiee ou corrompue non detectee | Empreintes SHA-256 et verification avec sha256sum -c |
| Serveur web trop generique | Confusion avec le site par defaut ou logs melanges | Vhost intranet dedie, site default desactive, logs separes |
| Processus web trop privilegie | Impact plus fort en cas de compromission | Worker Nginx execute sous www-nginx |
| Exposition SSH trop large | Surface d'attaque plus grande | UFW avec SSH restreint au sous-reseau 192.168.56.0/24 |
| Tentatives SSH repetees | Risque de force brute | Fail2ban actif sur la jail sshd |

## Procedure de restauration /var/www

Si /var/www est perdu :

1. Verifier la sauvegarde avec `sha256sum -c '/backup/www-checksums.txt'`.
2. Restaurer dans un repertoire de test avec `rsync -avz '/backup/www/' '/tmp/restauration-www-test/'`.
3. Controler les fichiers restaures avec `find '/tmp/restauration-www-test' -maxdepth 4 -type f`.
4. Restaurer en production seulement apres verification.
